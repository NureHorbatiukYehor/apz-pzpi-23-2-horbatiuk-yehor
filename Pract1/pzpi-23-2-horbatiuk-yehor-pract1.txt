3 ОПИС ВИКОНАНОЇ РОБОТИ
3.1 Проблема жорсткої залежності та сумісності продуктів
Під час проєктування складних програмних систем часто виникає ситуація, коли застосунок має оперувати сімействами взаємопов'язаних або залежних об'єктів (наприклад, елементами графічного інтерфейсу для різних операційних систем або компонентами генерації документів різних форматів). Наївна реалізація створення таких об'єктів зазвичай призводить до появи розлогої умовної логіки безпосередньо у клієнтському коді:Plaintext1  if (os == Windows) {
2      button = new WinButton();
3      checkbox = new WinCheckbox();
4  } else if (os == MacOS) {
5      button = new MacButton();
6      checkbox = new MacCheckbox();
7  }
Із розширенням системи (наприклад, додаванням підтримки Linux) виникають критичні архітектурні проблеми:Клієнтський код стає жорстко зв'язаним (tight coupling) з конкретними класами реалізації продуктів.Порушується принцип відкритості/закритості (Open/Closed Principle) SOLID — кожна нова платформа вимагає модифікації наявного клієнтського коду.Виникає ризик порушення сумісності, коли в системі випадково поєднуються об'єкти з різних родин (наприклад, WinButton та MacCheckbox), що призводить до нестабільної роботи програми.
3.2 Ідея патерна Абстрактна фабрика 
Патерн Абстрактна фабрика вирішує цю проблему шляхом інкапсуляції логіки створення сімейств об'єктів в окремі класи-фабрики та повної ізоляції клієнта від процесу інстанціювання за допомогою абстрактних інтерфейсів.Основні архітектурні принципи:Клієнтський код оперує виключно абстрактними інтерфейсами продуктів і фабрик.Створення кожного сімейства продуктів виділяється в окремий клас конкретної фабрики.Поєднання несумісних продуктів унеможливлюється на рівні архітектури, оскільки конкретна фабрика здатна генерувати продукти лише свого сімейства.Конфігурування застосунку конкретною фабрикою відбувається один раз під час ініціалізації.
3.3 Структура патерна та програмні контрактиАрхітектурна структура шаблона складається з наступних компонентів:AbstractProduct — інтерфейси, що визначають контракти для кожного типу продукту у сімействі.ConcreteProduct — специфічні реалізації продуктів під конкретні платформи чи конфігурації.AbstractFactory — базовий інтерфейс, який оголошує методи для створення всіх абстрактних продуктів.ConcreteFactory — класи, які реалізують методи базової фабрики та повертають екземпляри конкретних продуктів.Client — контекст, який використовує виключно абстрактні інтерфейси фабрик та продуктів.Мінімальна реалізація базових контрактів мовою Java виглядає наступним чином:Plaintext1  public interface Button {
2      void paint();
3  }
4  
5  public interface Checkbox {
6      void paint();
7  }
8  
9  public interface GUIFactory {
10     Button createButton();
11     Checkbox createCheckbox();
12 }
Логіка інкапсуляції процесу інстанціювання розподіляється за конкретними фабриками, які повертають базові інтерфейси замість конкретних реалізацій:Plaintext1  public class WinFactory implements GUIFactory {
2      public Button createButton() {
3          return new WinButton();
4      }
5      public Checkbox createCheckbox() {
6          return new WinCheckbox();
7      }
8  }
3.4 Переваги та недоліки патерна Абстрактна фабрика
Серед переваг патерна Абстрактна фабрика виділяють:Гарантована сумісність продуктів у межах одного сімейства.Усунення жорстких залежностей (Loose Coupling) між клієнтським кодом та конкретними реалізаціями.Дотримання принципу єдиної відповідальності (Single Responsibility Principle) — код створення продуктів зосереджено в одному місці.Полегшення тестування системи за рахунок можливості підміни реальних фабрик макетами (mock-об'єктами).До недоліків відносять:Збільшення загальної кількості класів у проєкті через необхідність створення окремих сутностей для кожного продукту та фабрики.Ускладнення підтримки коду при додаванні нового типу продукту до сімейства, що вимагає модифікації інтерфейсу AbstractFactory та всіх наявних конкретних фабрик
.3.5 Обмеження класичного патерна
Попри високу гнучкість при заміні сімейств, Абстрактна фабрика має суворе архітектурне обмеження: інтерфейс фабрики є фіксованим. Якщо у процесі еволюції системи виникає потреба розширити асортимент продуктів, це призводить до каскадних змін у всій ієрархії класів шаблона, що є трудомістким процесом у статично типізованих мовах.
3.6 Динамічний підхід на основі Прототипу (Prototype Factory)Для подолання обмеження жорсткого інтерфейсу класичної фабрики використовується альтернативний архітектурний підхід — фабрика на основі патерна Прототип. Замість створення розлогої ієрархії класів конкретних фабрик, створюється один універсальний клас фабрики, який параметризується прототипами об'єктів.Основні принципи динамічного підходу:Відсутність необхідності створювати класи WinFactory, MacFactory тощо.Фабрика зберігає асоціативний масив (map) або набір базових об'єктів-прототипів.Створення нового продукту відбувається шляхом клонування відповідного прототипу.Нові типи продуктів або нові сімейства можуть додаватися динамічно під час виконання програми (Runtime) без зміни наявних класів.Приклад динамічної концепції конфігурування фабрики:  
public class PrototypeFactory {
2      private Button buttonPrototype;
3      private Checkbox checkboxPrototype;
4  
5      public PrototypeFactory(Button b, Checkbox c) {
6          this.buttonPrototype = b;
7          this.checkboxPrototype = c;
8      }
9      public Button createButton() {
10         return (Button) buttonPrototype.clone();
11     }
12 }
3.7 Вимоги до мови для реалізації підходів
Для ефективної реалізації обох архітектурних підходів від мови програмування вимагаються певні інструменти:Класичний підхід вимагає наявності строгих інтерфейсів або абстрактних класів, підтримки поліморфізму підтипів та статичної типізації для виявлення помилок несумісності типів під час компіляції.Динамічний підхід на основі прототипів вимагає вбудованого механізму клонування об'єктів (наприклад, інтерфейсу Cloneable та методу clone() в Java) або гнучких засобів рефлексії (Reflection API) для інстанціювання об'єктів за їхнім типом під час виконання.
3.8 Порівняння класичної та динамічної фабрики
Критерій порівняння	Класична Абстрактна фабрика	Фабрика на основі Прототипу
Кількість класів у системі	Висока (потрібні класи для кожної фабрики)	Низька (достатньо одного класу фабрики)
Момент конфігурування	На етапі компіляції (Compile-time)	Під час виконання програми (Runtime)
Додавання нового продукту	Складне (вимагає зміни всіх інтерфейсів)	Легке (достатньо додати новий прототип)
Безпека типізації	Максимальна статична безпека	Часткова (залежить від реалізації clone)
